# DrawWork 稳定性优先开发方案

## 一、目标调整

本方案不以工期为主要约束，优先保证以下目标：

| 目标 | 要求 |
|------|------|
| 数据可靠性 | 协作数据不丢失，版本可恢复，关键写入有权限校验和审计 |
| 协作一致性 | 同一类数据采用同一套冲突处理策略，避免多人编辑时互相覆盖 |
| 安全边界 | 只暴露统一入口，内部服务不直接暴露到用户网段 |
| 性能可控 | 大画布、大文件、多人协作场景有明确限制、降级和监控 |
| 可运维性 | 部署、备份、恢复、升级、回滚都有可验证步骤 |
| 可用性 | 普通用户流程清晰，错误状态可理解，离线/断线有恢复路径 |

## 二、架构原则

### 2.1 对外入口

- 只对用户网段暴露 Nginx 的 `80/443`。
- PostgreSQL、Redis、Minio、Yjs、Adminer 默认只在 Docker 内网可访问。
- 管理工具通过 profile、堡垒机、SSH 隧道或临时白名单访问，不作为常驻公开端口。
- API、WebSocket、文件访问全部经过 Nginx，便于统一鉴权、限流、日志和 TLS。

### 2.2 数据一致性

| 数据类型 | 推荐策略 |
|----------|----------|
| Excalidraw 白板 | Yjs CRDT 为唯一在线编辑源，PostgreSQL 保存快照和版本 |
| 评论/投票/通知 | PostgreSQL 为事实源，Redis pub/sub 只做实时推送 |
| 思维导图/看板/泳道图 | 优先改为 Yjs 文档；如继续 JSONB 保存，必须加版本号/乐观锁 |
| 文件 | Minio 为文件源，PostgreSQL 保存元数据和权限归属 |

禁止让同一类可协作数据同时通过 Yjs 和整体 `PUT JSONB` 两条写入路径修改。

### 2.3 WebSocket 权限

- WebSocket 连接必须解析 `yjs_room_id` 到真实 `canvas_id` 和 `board_id`。
- 连接时校验 JWT、用户状态、画板权限和画布状态。
- 只有编辑者及所有者可建立可写 Yjs 连接。
- 查看者通过快照接口获取只读画面，避免只读用户通过 WebSocket 写入。
- 长期建议使用短期 WebSocket ticket，避免 JWT 长时间出现在 URL 查询参数里。

## 三、稳定性优先实施阶段

### Phase A: 基础安全与部署边界

- 修正 Nginx `/api/` 代理前缀，确保后端实际收到 `/api/...`。
- 移除 PostgreSQL、Redis、Minio、Yjs 的默认宿主机端口暴露。
- 生产环境启动时强制要求非默认 `JWT_SECRET`、数据库密码和 Minio 密钥。
- Adminer 仅通过 profile 临时启动，并限制来源。
- 增加上传类型、大小、文件数限制，上传必须绑定画板或画布权限。

### Phase B: 权限与接口一致性

- 抽象统一权限解析函数：用户、画板、画布、评论、投票都复用同一套权限等级。
- 所有基于 `canvas_id`、`comment_id`、`vote_id`、`file_id` 的接口都必须反查所属画板并校验权限。
- 分享 token 数据库存储 hash，只在创建响应中返回原始 token。
- 分享链接验证不消耗使用次数，真正接受/加入时再原子增加 `used_count`。
- Refresh Token 不允许直接访问普通 API，只能调用刷新接口换取 Access Token。

### Phase C: 协作持久化

- Yjs 服务根据 `yjs_room_id` 查询真实画布，不从字符串猜 `canvas_id`。
- 文档加载时只加载最新快照；保存时只保存脏文档，避免固定周期写入大量重复快照。
- 快照表增加 `snapshot_type`、`name`、`created_by`、`metadata`，支持自动/手动版本。
- 保留策略按数量和时间双维度执行，例如最近 50 个自动快照 + 所有手动快照。
- 恢复版本必须生成新的快照，不直接覆盖历史。

### Phase D: 文件与媒体

- 上传接口必须校验画板编辑权限。
- 文件访问通过 API 或 Nginx 内部代理完成权限校验，不直接暴露 Minio bucket。
- 存储对象名使用 UUID，保留原始文件名仅作展示。
- 文件类型同时校验 MIME、扩展名和大小。
- 视频/GIF 需要前端懒加载和暂停策略，避免大画布一次性播放所有媒体。

### Phase E: 性能与可观测性

- API 增加分页、排序、筛选和合理默认 limit。
- 高频写入接口增加限流和去抖，例如快照保存、评论、投票提交。
- 大画布增加元素数量、文件数量、单文件大小和总存储配额。
- 增加结构化日志，请求 ID 贯穿 Nginx、API、Yjs。
- 健康检查分层：`/health` 只检查进程，`/ready` 检查数据库、Redis、Minio。
- 压测场景至少覆盖：10/30/50 人同房间编辑、100MB 文件上传、断线重连、版本恢复。

### Phase F: 可用性与恢复

- 所有保存类操作提供明确状态：保存中、已保存、离线、重连中、保存失败。
- 断线期间本地可继续编辑，但恢复时必须明确同步状态。
- 查看者进入白板时展示只读状态，不显示可编辑工具。
- 删除画板/画布采用软删除，恢复和彻底删除分开。
- 备份恢复必须定期演练，并记录最近一次成功恢复时间。

## 四、当前代码审查重点

| 优先级 | 检查项 |
|--------|--------|
| P0 | Yjs 房间解析、画布权限、快照保存是否会写错 canvas |
| P0 | 上传接口是否任何登录用户都可上传任意文件 |
| P0 | 快照接口是否越权读取/写入 |
| P0 | Refresh Token 是否能当 Access Token 使用 |
| P1 | 分享 token 是否明文存储、验证是否消耗次数 |
| P1 | Nginx `/api/` 代理是否和 Express 路由一致 |
| P1 | 前端查看者是否仍可建立可写协作连接 |
| P2 | 结构化工具 JSONB 保存是否有乐观锁或 CRDT |
| P2 | Admin/TODO 接口是否影响产品验收范围 |

## 五、推荐验收门槛

- 后端单元/接口测试全部通过。
- 新增权限回归测试覆盖快照、上传、分享、投票。
- 前端构建通过，无明显控制台错误。
- Docker Compose 内只暴露 Nginx 端口。
- 使用两个不同权限用户手动验证：所有者可编辑，查看者只读且无法通过 API/WS 写入。
- 备份后恢复到新目录或新机器，能打开历史画板和媒体文件。
